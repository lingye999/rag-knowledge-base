知识库体系设计方案

基于腾讯云 LKEAP 文档解析能力 + 现有 AI 知识库产品设计 + agent-system 架构，构建 Agent 可用的知识库体系。
采用全 VDB 架构（腾讯云向量数据库 Dense+BM25 混合检索）+ PostgreSQL 权威数据源 + COS 对象存储三层存储，支持异步入库流水线与三维度加权检索。

一、系统整体架构

一、设计背景与目标

1.1 系统定位

知识库是独立子系统，为 Agent 提供企业文档的入库、检索、治理全链路能力。与记忆系统（短文本、用户维度、自动提取）互补，知识库面向长文档、租户维度、人工上传的场景。

维度知识库记忆系统
数据来源人工上传文档（PDF/Word/PPT/Excel/图片）Agent 对话中自动提取
数据粒度长文档 → 切片（300~500 字符）短文本片段（数十~数百 tokens）
隔离粒度租户 + 知识库维度用户维度
生命周期持久化，手动管理自动衰减（艾宾浩斯遗忘曲线）
检索方式Self-Querying + 混合检索 + 三维度加权意图检测 + BM25 混合检索

1.2 当前能力

系统已实现完整的入库 + 检索 + 治理链路：

能力实现方案
文档解析腾讯云 LKEAP 多模态解析（PDF/DOCX/PPTX/XLSX/图片→Markdown）
文本清洗4 阶段流水线（编码→格式噪声→内容噪声→双层输出）

检索引擎全 VDB 两路并行召回 + 三维度归一化加权（α0.7 + β0.1 + γ0.2）
异步入库PG SKIP LOCKED 队列 + Worker 协程池 + 指数退避重试
Agent 工具knowledge_search Tool，支持自然语言检索 + Schema 过滤

1.3 技术选型

架构范式: Provider + Factory 模式（StandaloneKnowledgeProvider 实现完整流水线）
解析层:   腾讯云 LKEAP（多模态文档解析 + Embedding + Rerank + QueryRewrite）
存储层:   PostgreSQL（权威数据源，9 张 ai_knowledge_* 表）
          + 腾讯云向量数据库（检索加速，kb_chunks + kb_doc_metadata 两集合）
          + COS 对象存储（原始文件 + 解析产物留档）
检索层:   全 VDB 架构（不回 PG），hybrid_search(dense 0.3 + sparse 0.7)
          + Self-Querying（LLM 从自然语言提取 Schema 过滤条件）
          + 三维度加权（α 相关性 + β 元数据 + γ 文档属性）
入库层:   7 阶段异步流水线（解析→清洗→打标→评分→切分→索引→元数据）
          PG SKIP LOCKED 队列 + Worker 协程池(N=4) + 指数退避重试
治理层:   19 字段元数据 Schema + 4 维质量评分 + 热度衰减 + VDB 健康巡检
隔离层:   tenant_id 全链路强制注入（PG WHERE + VDB FilterIndex + COS 分桶）

2.3 入库流水线（7 阶段）

文档上传后由 DocumentIngestionPipeline 异步处理，API 立即返回 task_id，后台 Worker 逐步执行：

阶段名称做什么耗时

Phase 1：文档解析

调用 LKEAP CreateReconstructDocumentFlow API 异步解析文档，完成后返回一个 ZIP 结果包的下载 URL。系统下载解压后按以下结构存储：

parsed/{doc_id}/
  ├── content.md          ← Markdown 全文（主产物，表格/公式/标题已结构化）
  ├── structure.json      ← 结构化 JSON（标题树、表格坐标、页面布局元信息）
  └── images/             ← 文档中提取的图片资源
        ├── img_1.png
        ├── img_2.png
        └── ...

提取优先级：.md → .json → .txt → .html（优先选最大的文件，避免空文件）

PG 状态回填：parse_status=parsed、parsed_md_url、parsed_json_url、page_count、total_chars

示例：上传一份 20 页的产品手册 PDF，LKEAP 返回的 Markdown 中表格被转为：

| 型号 | 量程 | 精度 |
|------|------|------|
| 3051S | 0~10MPa | ±0.025% |
| 3051C | 0~25MPa | ±0.04% |

Phase 2a：LLM 自动打标

输入文档前 4000 tokens + 租户的元数据 Schema，LLM 一次调用输出三类信息：

产出说明示例

归一化规则：LLM 输出值必须匹配 Schema 枚举值；用户手动指定的元数据优先级高于自动打标。

Phase 2b：质量评分

由 DocumentQualityScorer 对清洗后的文本进行 4 维度综合评估，输出 0.0~1.0 的质量分：

维度权重计算规则边界处理
完整度30% 1 - 失败页数 / 总页数 无分页概念的文件（TXT/MD）→ 直接得 1.0
结构度30% 标题覆盖率 × 0.5 + 段落长度合理性 × 0.5
  • 标题覆盖率：有 H1 → 0.4，有 H2 → +0.3，有 H3 → +0.3（满分 1.0）
  • 段落合理性：平均段落长度接近 100 字符为理想；<20 字符得 0.3，>500 字符得 0.5
 空文档 → 0.0；纯标题无正文 → heading×0.5
内容密度20% 非空白字符 / 总字符，达到理想值 0.7 得满分
低于 0.7 线性衰减：ratio=0.3 → 0 分，ratio=0.7 → 1.0 分		 空文档 → 0.0；密度 ≥0.7 → 1.0
清洗分20% 1 - clean_ratio（清洗掉的字符比例越低越好） 无清洗信号 → 保守给 1.0

综合公式：

quality_score = 0.3 × completeness + 0.3 × structure + 0.2 × density + 0.2 × clean_score

计算示例（一份 20 页产品手册 PDF）：

维度原始值得分加权贡献
完整度19/20 页解析成功0.950.3 × 0.95 = 0.285
结构度有 H1+H2+H3，平均段落 120 字符0.850.3 × 0.85 = 0.255
内容密度非空白占比 0.72（≥0.7 满分）1.000.2 × 1.00 = 0.200
清洗分clean_ratio=0.05（只清掉 5%）0.950.2 × 0.95 = 0.190
综合质量分0.93

预警与使用：

  • 低于 0.4 → 标记 quality_warning，检索时可选过滤
  • 质量分存入 PG ai_knowledge_document.quality_score
  • 分项信号存入 quality_signals JSON 字段（含 total_pages、content_chars、clean_ratio、各 dropped 计数）
  • 质量分同步到 VDB kb_doc_metadata.quality_score_x10k，参与检索 γ 维度加权（权重 0.1）

Phase 3：两级切分

直接按固定长度切会导致切片跨章节、句子被切断。两级切分解决这些问题：

第一级 — Segment（章节级）：按标题结构切成大段

  • 参数：min=3000 / target=4000 / max=6000 字符
  • 一级标题（# 、第X章）→ 强制分段
  • 二级标题（## 、第X节）→ 当前段 ≥3000 才分段

第二级 — Chunk（句子级）：在每个 Segment 内按句子累加

  • 参数：target=400 / min=100 / max=500 字符 / overlap=2 句
  • 按 。！？；.\n 切分句子，逐句累加到 target 后 flush
  • 相邻 Chunk 保留最后 2 句重叠，避免"断章取义"

示例（一份 15000 字符的产品手册）：

文档（15000 字符）
  ├── Segment 1: "第一章 产品介绍"（3200 字符）
  │     ├── Chunk 1: 产品概述...（400 字符）
  │     ├── Chunk 2: 技术优势...（380 字符）← 与 Chunk 1 有 2 句重叠
  │     └── Chunk 3: 应用场景...（420 字符）
  ├── Segment 2: "第二章 技术参数"（4100 字符）
  │     ├── Chunk 4: 量程范围...（390 字符）
  │     ├── Chunk 5: 精度等级...（410 字符）
  │     └── Chunk 6: 环境要求...（350 字符）
  └── Segment 3: "第三章 安装指南"（3500 字符）
        ├── Chunk 7: 安装准备...（400 字符）
        └── Chunk 8: 接线方法...（380 字符）

三、知识检索引擎

全 VDB 架构：所有检索信号都在向量数据库完成，命中后直接返回切片内容，不回 PG。

3.1 检索链路总览

3.2 Step 1 — 查询改写

多轮对话中，用户的当前问题可能包含指代词或省略。LLM 将其补全为独立可理解的检索查询：

改写能力示例
指代消解"那条件分支呢？"（上文提到审批流）→ "审批流里的条件分支怎么配置？"
缩写展开"3051参数" → "罗斯蒙特3051压力变送器技术参数规格"
同义词扩展"参数" → 追加 "技术规格 性能指标 量程"
噪音去除"请问帮我查一下" → 去除无检索价值的词

触发条件：有对话历史 + LLM 可用时触发；无历史或 LLM 不可用时跳过，直接用原始 query。

3.3 Step 2 — Self-Querying（意图识别）

从自然语言中自动提取元数据过滤条件，让用户无需手动选择分类：

输入: "制造业的成功案例"
输出: {
  "semantic_query": "成功案例",
  "filters": {"docCategory": "成功案例", "industryVertical": "制造业"}
}

输入: "3051压力变送器参数"
输出: {
  "semantic_query": "3051压力变送器参数",
  "filters": {}    ← 产品名称不做 filter，走语义检索
}

关键规则：宁可不过滤也不要错误过滤。只有用户明确说出分类词（"成功案例"、"制造业"）时才加 filter；产品名称、型号走语义检索，不做精确过滤。

3.4 Step 3 — 两路并行召回和定向召回

检索核心分为三路：路A 和路B 通过 asyncio.gather 并行执行（Phase 1），Phase 2 定向召回在路B完成后串行触发。三路各有不同的检索维度、参数和设计目的。

Dense 向量chunk.content embedding（1024d）文档 summary embedding（1024d）同路A

为什么这样设计

  • 路A 不带 filter：知识库文档打标不完整（LLM 自动打标可能遗漏），如果路A也带 filter，打标缺失的文档中的高质量切片会被完全排除。路A 不带 filter = 保底机制。
  • 路A 偏 BM25(0.7)：中文场景下产品型号（"3051"）、专业术语（"量程"）等关键词精确匹配比语义相似度更可靠，避免语义漂移。
  • 路B 带 filter：路B 的职责是"圈定正确文档范围"，filter 是其核心能力。即使 filter 过严导致 0 结果，有降级机制（去掉 filter 重试）兜底。
  • 路B 用 5 路加权 BM25：单集合只支持 1 路 sparse，通过文本重复拼接模拟字段 boost（title×3 权重最高），一次调用融合 ANN + BM25。
  • Phase 2 串行：依赖路B输出的 doc_id 列表作为搜索范围，无法与 Phase 1 并行。但搜索范围缩小到 50 个文档内，延迟很低（通常 <50ms）。
  • Phase 2 条件触发：纯语义查询（"3051参数"）不需要定向召回——路A全局搜已足够好，多一次 VDB 调用只增加延迟没有收益。

3.4.1 两路能力和定向召回差异对比

路 A（切片 hybrid）路 B（文档元数据 hybrid）Phase 2（定向召回）
实际作用 全局搜切片，不带 filter，返回 Top-30。按纯语义+BM25 排序，不感知分类意图。当用户说"制造业的产品手册"时，制造业文档的切片可能排在第 35、40 名——被 Top-30 截断了。路A 不知道用户想要"制造业"的东西。 带 filter 搜文档，返回 doc_id 列表。路B 知道哪些文档属于"制造业"，但它只返回 doc_id，不返回切片内容。它知道"答案在 docA 里"，但不知道 docA 的哪个切片最相关。 把路B的"文档范围"和路A的"切片搜索能力"结合：在路B圈定的 Top-50 文档内再搜一次切片，捞回路A因全局截断而遗漏的分类相关切片。
执行时机Phase 1 — 并行（asyncio.gather）Phase 2 — 串行（等路B完成后）
为什么并行/串行路A和路B互不依赖，搜不同集合，可同时发起依赖路B的输出（doc_id 列表）作为搜索范围，必须等路B完成
带 Filter❌ 不带（保底，防打标缺失漏召回）✅ 带（圈定分类范围）❌ 不带元数据 filter，用 doc_id IN 替代
触发条件始终执行始终执行仅当有 filter 且路B有结果时

3.5 Step 4-5 — 切片→文档聚合 + 三维度归一化加权

Step 3 的两路召回产出了两类原始数据，Step 4-5 将它们融合为每个文档的最终分数：

Step 3 产出：
  路 A → N 个切片，每个带 score（VDB hybrid_search 融合分）
  路 B → M 个 doc_id，按融合排名排序（同时作为 summary 和 meta_text 两路 RRF 输入）

Step 4-5 处理：
  路 A 的切片 → RRF 排名 → 按 doc_id 分组 → 几何衰减聚合 → 维度 α
  路 B 的 doc_id 排名 → RRF 贡献 × 0.15 → 维度 β
  VDB kb_doc_metadata 的属性字段 → 质量+时效+热度 → 维度 γ
  最终：finalScore = α×normA + β×normB + γ×normC → 映射到 [threshold, 1]

Step 4：切片→文档聚合（数据来源：路 A）

路 A 返回的是切片级分数，但三维度加权需要文档级分数。聚合步骤：

  1. RRF 排名转分数：将路 A 切片按 score 降序排列，每个切片的 RRF 贡献 = 1/(K + rank + 1)，K=60
  2. 按 doc_id 分组：同一文档的多个切片归为一组
  3. 几何衰减聚合：组内按 RRF 贡献降序，做 CHUNK_DECAY=0.2 的几何衰减求和
doc_rrf = seg[0] + seg[1]×0.2 + seg[2]×0.04 + seg[3]×0.008 + ...

示例（docA 有 3 个切片命中，rank 分别为 0、1、4）：
  chunk_01 rank=0 → rrf = 1/(60+0+1) = 0.01639
  chunk_12 rank=1 → rrf = 1/(60+1+1) = 0.01613
  chunk_45 rank=4 → rrf = 1/(60+4+1) = 0.01538

  docA_rrf = 0.01639 + 0.01613×0.2 + 0.01538×0.04 = 0.01703

为什么是几何衰减：前 3 个切片贡献 99%+，多切片命中有加分但不会喧宾夺主。

Step 5：三维度归一化加权

rawScore = α × normA + β × normB + γ × normC
finalScore = rawScore × (1 - threshold) + threshold    ← 映射到 [threshold, 1.0]
维度权重数据来源含义归一化方式
α — 相关性0.7路 A 切片 RRF → 文档聚合后的 doc_rrf切片内容与 query 的匹配程度理论锚点归一化：(doc_rrf - min) / (max - min) → [0, 1]
β — 元数据0.1路 B 文档排名（同一排名列表同时作为 summary 和 meta_text 两路 RRF 输入）文档标题/摘要/关键词与 query 的匹配两路 RRF 贡献 × 0.15 / 理论最大值 → [0, 1]
γ — 文档属性0.2VDB kb_doc_metadata 的 γ 属性字段（检索时一次返回）文档本身的静态质量信号三子项加和 / MAX_ATTR_BOOST → [0, 1]

维度 β 详解（数据来源：路 B 的 doc_id 排名）：

对每个文档，计算它在路 B 排名中的 RRF 贡献：
  boost = 1/(K_SUMMARY + rank + 1) + 1/(K_META_TEXT + rank + 1)
  boost × METADATA_WEIGHT(0.15)
  normB = boost / MAX_METADATA_BOOST

路 B hybrid_search 已融合 ANN + BM25，同一排名列表同时作为两路 RRF 输入。
示例：docA 在路 B 排名第 0 → boost = (1/61 + 1/61) × 0.15 = 0.00492 → normB = 1.0

维度 γ 三子项（数据来源：VDB kb_doc_metadata 属性字段，检索时直接读取不回 PG）：

子项权重VDB 字段公式含义
质量分0.10quality_score_x10k(quality/10000) × 0.1入库时 Phase 2b 评估的文档质量
时效性0.05date_published0.5^(age_ms/180天ms) × 0.05180 天半衰期指数衰减
热度0.05search_hit_countmin(1, log10(hit+1)/3) × 0.05被检索命中次数（定期同步）

最终映射：finalScore = rawScore × (1 - threshold) + threshold，将 [0,1] 的 rawScore 映射到 [threshold, 1.0]，确保通过 threshold 过滤的结果分数 ≥ threshold。

3.7 Step 6 — threshold 过滤

所有 finalScore < threshold 的结果被丢弃。阈值优先级：调用方显式传入 → KB 配置 min_score → 默认 0.3。

threshold行为适用场景
0.0关闭过滤数据少、宽松模式
0.3（默认）过滤明显无关常规 RAG
0.5中等严格专业问答
0.7宁缺毋滥法律/医疗

空结果是预期行为：让 LLM 知道"没答案"比"硬编答案"更有价值。

3.8 Step 7 — 后处理与返回

处理说明
HydrateVDB 直接返回切片 content 字段，不回 PG 查全文
Parent-Child 扩展按 chunk_index ± 1 从 VDB 拉前后相邻切片，拼接为 expanded_context
热度维护（异步）切片 hit_count +1、文档 search_hit_count +1，定期同步到 VDB
审计日志（异步）写 ai_knowledge_search_log（query/filters/命中/耗时/反馈）
